# 網路介面監控資料之 AIOps 教學設計  
## Python + Grafana 實作模組規劃

## 一、課程定位

本教學設計以「網路介面監控資料」為主軸，模擬類似 LibreNMS / RRDTool / SNMP 監控資料的時間序列情境，讓學員從原始 metrics 出發，逐步完成：

```text
時間序列特徵工程
→ Baseline Anomaly Detection
→ SPC for Anomaly Detection
→ Unsupervised ML for AD
→ Alert Reduction
→ Forecasting
→ Root Cause Analysis with LLM
```

整體設計目標是讓學員理解 AIOps 中的核心流程：

```text
Metrics → Features → Anomaly Detection → Alert → Alert Reduction → Forecasting → RCA → Action
```

## 二、使用的網路介面監控欄位

### 1. 流量大小：OCTETS

| 欄位 | 意義 |
|---|---|
| `INOCTETS` | 進入介面的位元組數 |
| `OUTOCTETS` | 離開介面的位元組數 |

用途：判斷流量大小、頻寬使用率、下載/上傳或資料傳輸壓力。

---

### 2. 封包數量：PACKETS

| 欄位 | 意義 |
|---|---|
| `INUCASTPKTS` | 進入介面的單播封包數 |
| `OUTUCASTPKTS` | 離開介面的單播封包數 |
| `INNUCASTPKTS` | 進入介面的非單播封包數 |
| `OUTNUCASTPKTS` | 離開介面的非單播封包數 |

用途：判斷封包數量、連線數、掃描行為、小封包問題。

---

### 3. 傳輸錯誤：ERRORS

| 欄位 | 意義 |
|---|---|
| `INERRORS` | 進入介面時發生錯誤的封包數 |
| `OUTERRORS` | 離開介面時發生錯誤的封包數 |

用途：判斷線路品質、port 問題、SFP / 光模組、NIC 或 duplex mismatch。

---

### 4. 丟棄封包：DISCARDS

| 欄位 | 意義 |
|---|---|
| `INDISCARDS` | 進入介面但被丟棄的封包數 |
| `OUTDISCARDS` | 離開介面時被丟棄的封包數 |

用途：判斷 queue / buffer 滿、頻寬壅塞、QoS 丟棄、設備處理能力不足。

---

### 5. Broadcast / Multicast

| 欄位 | 意義 |
|---|---|
| `INBROADCASTPKTS` | 進入介面的 broadcast 封包數 |
| `OUTBROADCASTPKTS` | 離開介面的 broadcast 封包數 |
| `INMULTICASTPKTS` | 進入介面的 multicast 封包數 |
| `OUTMULTICASTPKTS` | 離開介面的 multicast 封包數 |

用途：判斷 L2 loop、ARP storm、DHCP storm、broadcast storm、multicast flooding、IGMP 問題。

---

### 6. Unknown Protocol

| 欄位 | 意義 |
|---|---|
| `INUNKNOWNPROTOS` | 進入介面但無法識別協定的封包數 |

用途：判斷非標準協定、掃描、攻擊、設備設定錯誤或新服務導入。

---

## 三、資料模擬設計構想

### 1. 模擬資料格式

資料解析度設計為每 300 秒一筆，對應 RRD / SNMP polling 常見設定。

```text
timestamp, device_id, port_id,
INOCTETS, OUTOCTETS,
INERRORS, OUTERRORS,
INUCASTPKTS, OUTUCASTPKTS,
INNUCASTPKTS, OUTNUCASTPKTS,
INDISCARDS, OUTDISCARDS,
INUNKNOWNPROTOS,
INBROADCASTPKTS, OUTBROADCASTPKTS,
INMULTICASTPKTS, OUTMULTICASTPKTS
```

建議模擬期間：

```text
14 天～30 天，每 5 分鐘一筆
```

每個 port 可以產生一組時間序列，並加入不同異常事件。

---

### 2. 正常資料生成邏輯

正常網路流量應包含：

| 特性 | 模擬方式 |
|---|---|
| 日週期 | 白天流量高、晚上流量低 |
| 工作日/假日差異 | 工作日有較高業務流量 |
| 隨機波動 | 加入 Gaussian noise 或 lognormal noise |
| 突發小尖峰 | 模擬短時間正常業務尖峰 |
| In/Out 不對稱 | 依監控位置設定 inbound / outbound 差異 |

基本流量可以用：

```text
traffic(t) = baseline + daily_seasonality + weekly_effect + random_noise
```

封包數可由流量與平均封包大小反推：

```text
packets(t) ≈ octets(t) / average_packet_size
```

---

### 3. 模擬異常事件設計

#### Event A：正常業務流量增加

| 指標變化 | 說明 |
|---|---|
| `OCTETS ↑` |
| `UCASTPKTS ↑` |
| `DISCARDS ≈ 0` |
| `ERRORS ≈ 0` |

時間序列型態：流量與封包同步上升，但沒有錯誤與丟棄。

教學用途：區分「正常成長」與「故障異常」。

---

#### Event B：大量小封包 / 掃描

| 指標變化 | 說明 |
|---|---|
| `UCASTPKTS ↑↑` |
| `OCTETS 小幅↑` |
| `avg_packet_size ↓` |

時間序列型態：封包數暴增，但總流量不一定大。

可能原因：Port scan、DDoS 小封包、大量短連線、heartbeat 暴增。

---

#### Event C：大檔案傳輸 / 備份

| 指標變化 | 說明 |
|---|---|
| `OCTETS ↑↑` |
| `UCASTPKTS 中度↑` |
| `avg_packet_size ↑` |

時間序列型態：流量大幅上升，封包數中度上升。

可能原因：備份、資料同步、大檔下載、影音串流。

---

#### Event D：頻寬壅塞 / Queue 滿

| 指標變化 | 說明 |
|---|---|
| `OCTETS ↑↑` |
| `DISCARDS ↑` |
| `ERRORS ≈ 0` |

時間序列型態：高流量之後出現丟棄。

可能原因：頻寬不足、queue / buffer 滿、QoS policy 丟棄。

---

#### Event E：線路品質問題

| 指標變化            | 說明  |
| --------------- | --- |
| `ERRORS ↑`      |     |
| `DISCARDS 不一定↑` |     |
| `OCTETS 不一定高`   |     |

時間序列型態：`ERRORS` 從長期 0 變成 spike 或反覆 spike。

可能原因：線材、接頭、SFP、NIC、port、duplex mismatch。

---

#### Event F：高負載下線路不穩

| 指標變化 | 說明 |
|---|---|
| `OCTETS ↑` 後 `ERRORS ↑` |

時間序列型態：流量變高後，錯誤數才開始上升。

可能原因：高流量下暴露線材、port 或光模組品質問題。

---

#### Event G：Broadcast Storm / L2 Loop

| 指標變化 | 說明 |
|---|---|
| `BROADCAST ↑↑` |
| `NUCAST ↑↑` |
| 多 port 同時上升 |
| `DISCARDS` 可能上升 |

可能原因：Switch loop、STP 問題、ARP storm、DHCP storm、VLAN 過大。

---

#### Event H：Multicast Flooding

| 指標變化 | 說明 |
|---|---|
| `MULTICAST ↑↑` |
| 多 port 同時上升 |
| `DISCARDS` 可能上升 |

可能原因：IGMP snooping 未啟用、IGMP querier 問題、IPTV / routing protocol 異常。

---

#### Event I：異常設備大量發送

| 指標變化 | 說明 |
|---|---|
| 多個 `OUT*PKTS ↑` |
| `OUTOCTETS ↑` |
| 集中於某一 port |

可能原因：某台設備中毒、掃描、服務異常、備份排程錯誤。

---

#### Event J：Unknown Protocol

| 指標變化 | 說明 |
|---|---|
| `INUNKNOWNPROTOS ↑` |
| `INPKTS ↑` |

可能原因：非標準協定、掃描、攻擊、VLAN / routing 設定錯誤、新設備上線。

## 四、Notebook 1：Time Series Features

### 實作主題

```text
[實作] Python + Grafana：Notebook1 - Time Series Features
```

### 教學目標

學員能將原始 RRD / SNMP 監控欄位轉換成適合異常偵測的時間序列特徵。

### 核心概念

- Timestamp index
- Resampling
- Rolling statistics
- Lag features
- Rate / ratio features
- Inbound vs outbound comparison
- 平均封包大小
- error rate / discard rate

### 實作內容

1. 讀取模擬或匯出的 RRD metrics CSV
2. 整理 timestamp 與欄位型態
3. 處理 `NaN`
4. 建立 rolling features：

```text
rolling_mean_1h
rolling_std_1h
rolling_median_1h
rolling_p95_1d
rolling_mad_1d
```

5. 建立衍生指標：

```text
traffic_total = INOCTETS + OUTOCTETS
ucast_total = INUCASTPKTS + OUTUCASTPKTS
errors_total = INERRORS + OUTERRORS
discards_total = INDISCARDS + OUTDISCARDS
broadcast_total = INBROADCASTPKTS + OUTBROADCASTPKTS
multicast_total = INMULTICASTPKTS + OUTMULTICASTPKTS
avg_packet_size = traffic_total / ucast_total
error_rate = errors_total / ucast_total
discard_rate = discards_total / ucast_total
broadcast_ratio = broadcast_total / ucast_total
multicast_ratio = multicast_total / ucast_total
```

### Grafana 對應展示

- 流量 In / Out 趨勢圖
- 封包數趨勢圖
- error / discard rate panel
- avg packet size panel
- broadcast / multicast trend panel

### 預期產出

- `features.csv`
- Time series feature dashboard
- 學員能說明每個 feature 的監控意義

### 教學目標

- Static threshold

### 實作內容

1. 對 `traffic_total`, `ucast_total`, `errors_total`, `discards_total` 建立 baseline
2. 計算 anomaly score：

```text
z_score = (x - rolling_mean) / rolling_std
robust_z = (x - rolling_median) / MAD
percentile_exceed = x > rolling_p95
```

3. 將異常轉成 flags：

```text
traffic_high
packets_high
errors_high
discards_high
broadcast_high
multicast_high
unknown_high
```

4. 比較不同 baseline 方法對 spike / plateau / trend 的效果

### Grafana 對應展示

- 原始值 + baseline band
- anomaly flag time series
- 異常事件 table panel
- annotation：標註模擬事件發生時間

### 預期產出

- Event aggregation
- Temporal merging
- Deduplication
- Severity scoring
- Suppression

### 實作內容

2. 設計 alert aggregation rule：


## 十、Notebook 7：Root Cause Analysis with LLM

### 實作主題

```text
[實作] Python + Grafana ：Notebook7 - Root Cause Analysis with LLM
```

### 教學目標

將多欄位異常、時間序列 pattern、alert reduction 結果與 context 整合，產生 RCA 診斷摘要與建議處置。

### 核心概念

- RCA knowledge graph
- Evidence-based reasoning
- Metric interaction rules
- Root cause candidate ranking
- LLM-assisted explanation
- Runbook generation
- Metrics + logs context

### RCA 基本流程

```text
RRD Metrics
→ Time Window
→ Anomaly Flags
→ Metric Interaction Rules
→ Root Cause Candidate Ranking
→ LLM Explanation
→ Recommended Action / Runbook
```

### 實作內容

1. 讀入 reduced alerts
2. 彙整 event window 的 evidence：

```text
affected_port
time_window
abnormal_metrics
pattern_type
duration
magnitude
related_ports
```

3. 建立 RCA rule table：

| 條件 | RCA Candidate |
|---|---|
| `OCTETS ↑↑ + DISCARDS ↑ + ERRORS ≈ 0` | Queue congestion |
| `ERRORS ↑ + DISCARDS 不一定↑` | Link quality issue |
| `UCASTPKTS ↑↑ + OCTETS 小幅↑` | Small packet / scanning |
| `BROADCAST ↑↑ + 多 port 同時↑` | Broadcast storm / L2 loop |
| `MULTICAST ↑↑ + 多 port 同時↑` | Multicast flooding |
| `INUNKNOWNPROTOS ↑ + INPKTS ↑` | Unknown protocol / scan |

4. 建立 root cause score：

```text
score = evidence_match_weight + anomaly_magnitude + duration_weight + affected_ports_weight
```

5. 產生 LLM prompt：

```text
你是一位 AIOps / Network RCA 助理。
請根據以下監控證據判斷可能根因，並輸出：
1. 問題型態
2. 主要證據
3. 可能根因
4. 建議處置
5. 信心分數
```

6. 產生 RCA report：

```text
事件時間：2026-02-25 18:10–18:30
影響介面：port-id7434
問題型態：Queue congestion / buffer pressure

主要證據：
- OUTOCTETS 高於 rolling p95
- OUTDISCARDS 同步上升
- OUTERRORS 維持 0
- UCASTPKTS 同步上升

可能根因：
- 高流量造成 queue / buffer 壓力
- 可能與備份、資料同步或 QoS 設定有關

建議處置：
- 檢查 Top Talker
- 檢查 QoS / queue 設定
- 檢查該時間是否有備份或同步任務
- 若長期發生，評估擴充頻寬
```

### Grafana 對應展示

- RCA event table
- evidence panel
- root cause candidate ranking
- LLM explanation text panel
- runbook link / action panel

### 預期產出

- `rca_events.csv`
- `rca_report.md`
- RCA dashboard
- 學員理解從 AD 到 RCA 的完整流程

4. 理解單欄位異常與多欄位交互作用的差異

6. 使用 forecasting 進行 early warning
7. 利用多欄位證據進行 RCA

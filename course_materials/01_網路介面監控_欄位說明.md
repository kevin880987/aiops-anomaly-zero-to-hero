# 網路介面監控欄位說明

## 1. 資料背景

此類資料通常來自網路設備的 SNMP 介面統計，例如 switch、router、firewall、server NIC 等。若資料來自 RRD 檔案，通常表示系統以固定時間間隔，例如每 300 秒，記錄一次網路介面的流量、封包、錯誤與丟棄狀況。

這些欄位可以視為 AIOps / AI-SPC 的 Layer 1 Metrics 資料來源，適合用於：

- 網路流量監控
- 異常偵測
- 頻寬壅塞分析
- 封包丟棄分析
- 線路品質監控
- Broadcast / Multicast 問題診斷
- 根本原因分析 RCA

---

## 2. 欄位總覽

| 類別 | 欄位 | 全名 / 意義 | 方向 | 單位 | 主要用途 |
|---|---|---|---|---|---|
| 流量大小 | `INOCTETS` | Input Octets，流入該介面的位元組數 | In / RX | Bytes | 看進入介面的資料量 |
| 流量大小 | `OUTOCTETS` | Output Octets，流出該介面的位元組數 | Out / TX | Bytes | 看離開介面的資料量 |
| 錯誤 | `INERRORS` | 接收方向錯誤封包數 | In / RX | Packets | 看接收時的傳輸錯誤 |
| 錯誤 | `OUTERRORS` | 傳送方向錯誤封包數 | Out / TX | Packets | 看傳送時的傳輸錯誤 |
| 單播封包 | `INUCASTPKTS` | 接收方向 unicast 封包數 | In / RX | Packets | 看一對一封包量 |
| 單播封包 | `OUTUCASTPKTS` | 傳送方向 unicast 封包數 | Out / TX | Packets | 看一對一封包量 |
| 非單播封包 | `INNUCASTPKTS` | 接收方向 non-unicast 封包數 | In / RX | Packets | Broadcast + Multicast 總量 |
| 非單播封包 | `OUTNUCASTPKTS` | 傳送方向 non-unicast 封包數 | Out / TX | Packets | Broadcast + Multicast 總量 |
| 丟棄 | `INDISCARDS` | 接收方向被丟棄的封包數 | In / RX | Packets | 看 queue / buffer / QoS 問題 |
| 丟棄 | `OUTDISCARDS` | 傳送方向被丟棄的封包數 | Out / TX | Packets | 看 queue / buffer / QoS 問題 |
| 未知協定 | `INUNKNOWNPROTOS` | 收到但設備無法辨識協定的封包數 | In / RX | Packets | 看協定異常、掃描或設定錯誤 |
| Broadcast | `INBROADCASTPKTS` | 接收方向 broadcast 封包數 | In / RX | Packets | 看 ARP / DHCP / broadcast storm |
| Broadcast | `OUTBROADCASTPKTS` | 傳送方向 broadcast 封包數 | Out / TX | Packets | 看 ARP / DHCP / broadcast storm |
| Multicast | `INMULTICASTPKTS` | 接收方向 multicast 封包數 | In / RX | Packets | 看 IGMP / IPTV / routing protocol |
| Multicast | `OUTMULTICASTPKTS` | 傳送方向 multicast 封包數 | Out / TX | Packets | 看 IGMP / IPTV / routing protocol |

## 3. In / Out 的正確解讀

`In` 和 `Out` 是以「被監控的那個 interface」為視角，不是以使用者主觀的下載 / 上傳為視角。

- `INOCTETS`：流入該 interface 的資料量。
- `OUTOCTETS`：流出該 interface 的資料量。

因此「下載」不一定永遠對應 `INOCTETS`，要看你監控的是哪個 port：

- 若監控 WAN-facing interface，外部資料進入設備，可能是 `INOCTETS` 上升。
- 若監控 LAN-facing interface，資料從設備送往內部使用者，可能是 `OUTOCTETS` 上升。

教學上應避免直接說：

```text
INOCTETS = 下載
OUTOCTETS = 上傳
```

比較嚴謹的說法是：

```text
INOCTETS = 流入該監控介面的位元組數
OUTOCTETS = 流出該監控介面的位元組數
```

---

## 4. Octets 與頻寬換算

Octets 幾乎可以視為 Bytes。若要換算成 bits per second 或 Mbps：

```text
bits = octets × 8
Mbps = octets × 8 / 1,048,576
```

若 RRD 中的數值已經是 rate，例如每秒 octets，則：

```text
Mbps = octets_per_sec × 8 / 1,048,576
```

例如：

```text
INOCTETS = 854330.3240
Mbps = 854330.3240 × 8 / 1,048,576 ≈ 6.52 Mbps
```

## 5. 流量與封包的關係

核心概念：

```text
流量大小 ≈ 封包數量 × 平均封包大小
OCTETS ≈ PACKETS × Average Packet Size
```

- `OCTETS` 看資料量。
- `PACKETS` 看封包數量。
- `Average Packet Size` 看平均每個封包有多大。

| 現象 | 可能解讀 |
|---|---|
| `OCTETS ↑` + `PACKETS ↑` | 正常流量增加、服務使用量上升 |
| `OCTETS ↑↑` + `PACKETS 中度↑` | 大檔案傳輸、備份、影音串流 |
| `PACKETS ↑↑` + `OCTETS 小幅↑` | 大量小封包、掃描、攻擊、heartbeat、短連線 |
| `OCTETS ↑` + `PACKETS ↑` + `DISCARDS ↑` | 可能壅塞或 queue 滿 |
| `OCTETS ↓` + `PACKETS ↓` | 低使用量、服務中斷或連線異常 |

類比說法：

```text
OCTETS 高 = 車上載很多貨
PACKETS 高 = 車輛數很多
```

## 6. Broadcast 與 Multicast

### Broadcast

Broadcast 是「一對所有」。同一 VLAN 或區網內的所有設備都會收到。

常見例子：

- ARP
- DHCP Discover
- NetBIOS / legacy discovery

異常時可能代表：

- Broadcast storm
- ARP storm
- DHCP storm
- Switch loop
- STP 問題
- VLAN 設計過大

### Multicast

Multicast 是「一對多」，只送給加入特定 multicast group 的設備。

常見例子：

- IPTV / video streaming
- Routing protocol，例如 OSPF
- mDNS / SSDP
- Cluster / HA heartbeat

異常時可能代表：

- Multicast flooding
- IGMP snooping 問題
- IGMP querier 問題
- IPTV / streaming 設定異常
- Routing protocol 異常

## 7. 快速判斷口訣

```text
OCTETS 看流量大小
PACKETS 看封包數量
ERRORS 看傳輸品質
DISCARDS 看壅塞與 queue
BROADCAST / MULTICAST 看二層網路與群播問題
UNKNOWNPROTOS 看協定異常
```
